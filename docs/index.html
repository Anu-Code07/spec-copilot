Frontend spec-driven development

Kiro-style specs for Flutter, Next.js, and React Native. From feature idea → requirements → gap analysis → UI design → implementation tasks.

Get started View on GitHub

Why SpecDrive?

Backend teams have OpenAPI and Spec Kit. Frontend developers need the same structure — but focused on screens, components, navigation, state, and accessibility — not REST APIs.

📋

Kiro-style gates

Approve requirements, gap-analysis, design, and tasks before implementation.

🔍

Gap analysis

Compare requirements against your existing codebase before writing design.md.

🎨

design.md drives UI

Screens, components, flows, state management, and a11y — the frontend blueprint.

🤖

AI-agnostic

CLI with free LLM, or MCP in Cursor/Claude using your host AI keys.

Workflow

Feature idea → requirements.md approve ✓ → gap-analysis.md approve ✓ → design.md approve ✓ → tasks.md approve ✓ → Implement → Review

Quick install

# In your frontend project
npm install -g @specdrive/cli

spec init --stack flutter
spec create "Product Review Screen"

# With free LLM (CLI mode)
export GEMINI_API_KEY="your-key"
spec gap-analysis --spec product-review-screen

See Getting started for the full gated workflow.

CLI vs MCP

Mode Who generates docs API keys
CLI / npm SpecDrive free LLM chain GEMINI_API_KEY, GROQ_API_KEY, or Ollama
MCP (Cursor / Claude) Host AI (Cursor / Claude) Your editor's API key — SpecDrive never calls LLM
